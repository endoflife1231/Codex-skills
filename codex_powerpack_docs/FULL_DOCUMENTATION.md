# Полная документация: единый дистрибутив Codex Powerpack

## 1. Контекст проекта

Рабочая среда пользователя:

- GitHub Codespaces;
- проект открыт через настольный VS Code;
- Codex используется как основной агент разработки;
- пользователь хочет переносимую конфигурацию, которую можно установить в разные локальные проекты.

Начальная идея состояла в том, чтобы перенести в Codex полезный стиль работы Claude-подобных агентных систем.

Под «надстройкой» понимаются:

- инструкции;
- поверхностная настройка;
- порядок анализа;
- планирование;
- самопроверка;
- правила использования инструментов;
- разделение работы между ролями;
- требования к итоговому ответу.

### Главное ограничение

Нельзя установить интеллект одной модели поверх другой.

Codex останется Codex. Можно изменить только то, как он организует работу.

Хорошая настройка способна сделать работу более надёжной, но не превращает модель в другую модель.

---

## 2. Какой практический эффект ожидается

Надстройка должна улучшить не «чистый интеллект», а дисциплину выполнения задачи.

Ожидаемый эффект:

- сначала изучить проект, а затем менять код;
- не редактировать лишние файлы;
- отделять исследование от реализации;
- составлять план для сложных задач;
- проверять предположения;
- запускать тесты;
- проводить независимое ревью;
- явно сообщать, что не удалось проверить;
- аккуратнее работать с большими репозиториями;
- сохранять важные технические детали в отчёте.

### Где эффект больше

- крупный рефакторинг;
- архитектурные изменения;
- сложная отладка;
- авторизация;
- платежи;
- миграции базы данных;
- безопасность;
- изменения сразу в frontend, backend и инфраструктуре;
- незнакомый большой проект.

### Где эффект меньше

- переименование;
- простая кнопка;
- небольшая правка текста;
- очевидная однострочная ошибка;
- простой скрипт.

Для этого нужны разные режимы и профили, чтобы сложная оркестрация не мешала простым задачам.

## 3. Субагенты

Субагенты — это отдельные специализированные помощники внутри общего процесса.

Пример:

- Explorer изучает проект;
- Planner формирует план;
- Implementer пишет код;
- Tester запускает проверки;
- Reviewer ищет ошибки;
- Security Reviewer проверяет риски;
- главный агент собирает результаты.

### Зачем нужны субагенты

Один агент может пропускать ошибки, потому что одновременно:

- изучает;
- планирует;
- пишет;
- проверяет;
- объясняет.

Разделение ролей уменьшает смешивание задач.

### Базовый набор

#### Explorer

- читает проект;
- ищет точки входа;
- определяет зависимости;
- не меняет файлы.

#### Planner

- формирует последовательность действий;
- определяет риски;
- не меняет файлы.

#### Implementer

- вносит минимальные изменения;
- следует согласованному плану;
- запускает локальные проверки.

#### Debugger

- воспроизводит ошибку;
- ищет первопричину;
- не маскирует симптомы.

#### Reviewer

- проверяет diff;
- ищет регрессии;
- оценивает качество решения;
- по умолчанию не редактирует файлы.

#### Tester

- определяет необходимые тесты;
- запускает их;
- сохраняет команды и результаты.

#### Security Reviewer

- проверяет права доступа;
- ввод пользователя;
- секреты;
- сетевые вызовы;
- зависимости;
- необратимые действия.

#### Architect

- проверяет архитектурные последствия;
- ищет лишнюю связанность;
- оценивает границы модулей.

### Правило записи

По умолчанию только один исполнитель должен изменять один и тот же набор файлов.

Исследование и ревью можно выполнять параллельно. Параллельное редактирование одинаковых файлов создаёт конфликты.

## 4. Источники компонентов

Пользователь планирует предоставить:

1. репозиторий с Claude-подобными системными инструкциями;
2. репозиторий с субагентами;
3. собственный большой Skill-пак под Codex;
4. Caveman;
5. Graphify;
6. собственные правила проекта.

### Что делать с первым источником

Не копировать всё подряд.

Нужно разделить инструкции на категории:

- универсальные;
- Claude-специфичные;
- относящиеся к отсутствующим инструментам;
- подходящие для `AGENTS.md`;
- подходящие для Skills;
- подходящие для конкретных субагентов;
- справочные;
- конфликтующие;
- устаревшие;
- опасные.

### Что делать со вторым источником

Не устанавливать всех агентов автоматически.

Для каждого агента определить:

- назначение;
- необходимость;
- формат Codex;
- права;
- доступ к записи;
- инструменты;
- формат результата;
- зависимости;
- лицензию.

### Что делать со Skill-паком

Пакет уже большой, поэтому нужен реестр и профили.

Слишком большое количество активных Skills может:

- усложнить выбор;
- создавать конфликты;
- дублировать назначения;
- перегружать стартовое описание доступных возможностей.

Поэтому Skills должны включаться выборочно.

---

## 5. AGENTS.md, Skills, субагенты и references

Нельзя помещать все инструкции в один огромный `AGENTS.md`.

### В AGENTS.md

Только короткие постоянные правила:

- сначала исследовать;
- не менять лишнее;
- проверять реальные файлы;
- запускать тесты;
- сообщать ограничения;
- использовать субагентов по необходимости;
- не доверять графу без проверки;
- не скрывать критические ошибки;
- не выполнять опасные команды без явного основания.

### В Skills

Длинные процессы:

- полное code review;
- систематическая отладка;
- планирование функции;
- проверка перед завершением;
- миграция;
- безопасность;
- выпуск релиза;
- работа с Graphify;
- Caveman.

### В субагентах

- роль;
- область ответственности;
- права;
- ограничения;
- формат ответа;
- критерии завершения.

### В references

- большие чек-листы;
- подробные соглашения;
- длинные пояснения;
- справочные документы.

---

## 6. Caveman

Caveman используется для оптимизации вывода.

Он должен уменьшать:

- повторения;
- вводные фразы;
- очевидные объяснения;
- излишне длинные отчёты.

Он не должен удалять:

- команды;
- ошибки;
- stack trace;
- имена файлов;
- номера строк;
- список тестов;
- предупреждения безопасности;
- незавершённые риски;
- информацию об откате;
- факты о непроверенных частях.

### Рекомендуемая политика

- главный агент: Caveman Lite;
- финальный ответ: Lite или Full;
- Explorer: без агрессивного сжатия;
- Planner: умеренное сжатие;
- Tester: не сокращать команды и результаты;
- Reviewer: сохранять все замечания;
- Security Reviewer: не применять агрессивное сжатие;
- Debugger: не сокращать данные воспроизведения.

Caveman должен быть отключаемым компонентом.

---

## 7. Graphify

Graphify нельзя считать обычным текстовым Skill.

Обычно подобный компонент включает:

- исполняемый движок;
- зависимости;
- команды установки;
- команды анализа;
- выходные файлы;
- hooks;
- конфигурацию;
- обновление;
- удаление.

### Роль Graphify

Graphify помогает:

- увидеть связи между файлами;
- исследовать архитектуру;
- найти затронутые модули;
- ориентироваться в большом коде;
- уменьшить бессистемное чтение репозитория.

### Ограничение

Graphify — вспомогательный индекс, а не абсолютная истина.

Перед изменением кода агент обязан открыть реальные исходные файлы.

Граф может быть:

- устаревшим;
- неполным;
- неверно построенным;
- ограниченным языком;
- построенным до последних изменений.

### Интеграция

Graphify должен иметь отдельный адаптер:

```text
integrations/graphify/
├── install.sh
├── run.sh
├── update.sh
├── doctor.sh
├── uninstall.sh
├── adapter.json
└── default.graphifyignore
```

Точные команды нельзя придумывать заранее. Они определяются после изучения предоставленного исходника.

### Graphify и Caveman

Они решают разные задачи:

- Graphify помогает получать контекст;
- Caveman сокращает итоговый вывод.

Они могут работать вместе.

## 9. Каталог исходных данных

До сборки пользователь создаёт:

```text
sources/
├── claude-overlay/
├── subagents/
├── skills-pack/
├── caveman/
├── graphify/
└── project-rules/
```

Для каждого источника нужны:

- URL;
- локальный путь;
- commit или tag;
- лицензия;
- дата получения;
- назначение;
- разрешённые для использования файлы;
- запрещённые для копирования файлы;
- заметки.

## 10. Конвейер сборки

### Этап 1. Инвентаризация

Codex должен:

- найти все источники;
- прочитать документацию;
- найти лицензии;
- найти исполняемые файлы;
- найти hooks;
- определить зависимости;
- определить форматы;
- найти дубликаты;
- найти конфликтующие инструкции;
- найти опасные команды.

Результат:

```text
build/reports/source-inventory.md
build/source-manifest.json
```

### Этап 2. Проектирование

Codex создаёт:

```text
build/reports/implementation-plan.md
```

В плане:

- mapping источников;
- выбранные агенты;
- исключённые агенты;
- выбранные Skills;
- конфликты;
- стратегия AGENTS.md;
- стратегия Caveman;
- стратегия Graphify;
- тестовая матрица;
- лицензии;
- Codespaces.

### Этап 3. Нормализация

- UTF-8;
- единые переносы строк;
- удаление `.git`;
- исключение caches;
- сохранение лицензий;
- фиксация версии;
- вычисление hashes.

### Этап 4. Адаптация инструкций

- удалить ссылки на отсутствующие инструменты;
- заменить специфичные команды;
- не менять смысл ограничений;
- разделить постоянные и редкие инструкции;
- убрать дублирование;
- сократить основной контекст.

### Этап 5. Адаптация субагентов

Для каждого агента создать карточку:

```text
name
purpose
when_to_use
read_permissions
write_permissions
tools
output_format
completion_criteria
caveman_policy
graphify_policy
source
license
```

### Этап 6. Интеграция Skills

- проверить `SKILL.md`;
- построить registry;
- найти дубликаты;
- назначить профиль;
- записать зависимости;
- добавить Caveman;
- добавить Graphify wrapper.

### Этап 7. Graphify

- изучить реальный способ установки;
- создать адаптер;
- добавить doctor;
- добавить конфигурацию;
- обеспечить отключение;
- продумать актуальность графа;
- проверить Codespaces.

### Этап 8. Установщик

- backup;
- dry-run;
- merge;
- state manifest;
- verify;
- update;
- uninstall;
- повторный запуск без дублирования.

### Этап 9. Тестирование

- чистый проект;
- существующий AGENTS.md;
- повторная установка;
- обновление;
- удаление;
- профили;
- Graphify включён и выключен;
- Caveman включён и выключен;
- отсутствующие зависимости;
- Codespaces;
- восстановление backup.

### Этап 10. Выпуск

```text
release/codex-powerpack-X.Y.Z.zip
release/codex-powerpack-X.Y.Z.sha256
release/codex-powerpack-X.Y.Z-manifest.json
release/codex-powerpack-X.Y.Z-test-report.md
```

## 11. Профили установки

### Minimal

- core;
- Explorer;
- Planner;
- Reviewer;
- Tester;
- базовые Skills;
- Caveman;
- Graphify опционально.

### Standard

- всё из Minimal;
- Implementer;
- Debugger;
- Architect;
- Security Reviewer;
- расширенные Skills;
- Graphify по умолчанию, если совместим.

### Full

- все проверенные компоненты;
- дополнительные агенты;
- специализированные Skills;
- расширенная диагностика.

Full не означает «включить всё найденное». Непроверенные компоненты не включаются.

---

## 12. Установщик

Предлагаемая команда:

```bash
bash install.sh   --target /workspaces/my-project   --profile standard   --with-graphify   --backup
```

### Параметры

```text
--target PATH
--profile minimal|standard|full
--mode project|user
--with-graphify
--without-graphify
--with-caveman
--without-caveman
--dry-run
--backup
--force
--non-interactive
--verbose
```

### Поведение

1. Проверить путь.
2. Проверить права.
3. Найти существующие настройки.
4. Проверить зависимости.
5. Показать план.
6. Создать backup.
7. Установить компоненты.
8. Объединить AGENTS.md.
9. Создать state manifest.
10. Запустить verify.
11. Показать отчёт.

### Идемпотентность

Повторный запуск не должен:

- дублировать блоки;
- создавать копии одинаковых агентов;
- повторно добавлять Skills;
- ломать пользовательские изменения.

---

## 13. Управляемые блоки AGENTS.md

Нельзя полностью перезаписывать существующий файл.

Используются маркеры:

```markdown
<!-- CODEX-POWERPACK:BEGIN core -->
Управляемые правила
<!-- CODEX-POWERPACK:END core -->
```

Отдельные блоки:

- core;
- graphify;
- caveman;
- project-rules.

Обновляется только содержимое внутри маркеров.

---

## 14. State manifest

После установки:

```text
.codex-powerpack/state.json
```

Содержимое:

- версия;
- дата;
- профиль;
- режим;
- список файлов;
- hashes;
- источники;
- версии;
- backup;
- интеграции;
- изменения AGENTS.md;
- команды диагностики.

Uninstall должен удалять только то, что перечислено в state manifest.

---

## 15. GitHub Codespaces

Внешние инструменты могут исчезать после rebuild контейнера.

Нужно:

- не перезаписывать `.devcontainer.json`;
- создать snippet;
- создать restore-скрипт;
- предложить `postCreateCommand`;
- проверить инструмент после rebuild.

Пример идеи:

```json
{
  "postCreateCommand": "bash .codex-powerpack/restore-tools.sh"
}
```

Фактическое изменение пользователь добавляет осознанно.

---

## 16. Безопасность

Потенциально опасны:

- hooks;
- shell-скрипты;
- сетевые установки;
- сторонние Skills;
- агенты с записью;
- команды удаления;
- prompt injection в документации проекта;
- зависимости.

Обязательные правила:

- не запускать неизвестные hooks;
- не использовать `curl | sh`;
- закреплять версии;
- проверять checksum;
- не читать секреты без необходимости;
- не коммитить `.env`;
- не логировать токены;
- минимизировать права записи;
- проверять shell-код;
- использовать safe quoting;
- тестировать uninstall.

---

## 17. Лицензии

До включения каждого источника нужно проверить:

- разрешено ли использование;
- разрешено ли изменение;
- разрешено ли перераспространение;
- требуется ли attribution;
- нужно ли сохранить copyright;
- совместимы ли лицензии.

Если перераспространение не подтверждено:

- не включать исходник;
- использовать локальный путь;
- установить отдельно;
- создать адаптер;
- дать ручную инструкцию.

---

## 18. Тестирование

### Установка

- чистый проект;
- существующий AGENTS.md;
- существующие агенты;
- существующие Skills;
- повторная установка;
- dry-run;
- все профили.

### Graphify

- доступен;
- отсутствует;
- установка отключена;
- зависимость отсутствует;
- граф устарел;
- doctor обнаруживает ошибку.

## 19. Критерии готовности

Дистрибутив готов, если:

- устанавливается в чистый проект;
- не ломает существующие настройки;
- поддерживает профили;
- Codex видит агентов и Skills;
- Caveman работает по политике;
- Graphify работает или корректно отключается;
- verify ловит ошибки;
- update сохраняет пользовательские изменения;
- uninstall безопасен;
- архив воспроизводим;
- лицензии приложены;
- тестовый отчёт пройден.

## 20. Что нельзя делать

- Не копировать все инструкции в AGENTS.md.
- Не устанавливать всех агентов без отбора.
- Не включать все Skills без реестра.
- Не давать всем агентам запись.
- Не доверять Graphify без чтения кода.
- Не сокращать критические отчёты Caveman.
- Не перезаписывать пользовательские файлы.
- Не использовать плавающие версии.
- Не запускать hooks автоматически.
- Не включать исходник без лицензии.
- Не обещать перенос интеллекта другой модели.
